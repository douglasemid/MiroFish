Visão geral do projeto

O que é o MiroFish

MiroFish é um motor de simulação de inteligência coletiva baseado em multi-agentes LLM. A partir de um material-semente (PDF, MD, TXT), ele:

  1. Extrai entidades, gera personas com personalidade, memória de longo prazo e lógica própria
  2. Constrói automaticamente um "mundo digital paralelo" usando o framework OASIS (CAMEL-AI) que simula plataformas estilo Twitter e Reddit
  3. Coloca centenas/milhares de agentes para interagir nesse mundo durante N rodadas de tempo simulado
  4. Um Report Agent analisa toda a simulação e devolve um relatório de previsão estruturado

Tradução completa para PT-BR

O MiroFish nativo só tem dois idiomas de UI traduzidos: Chinês (zh) e Inglês (en). Para o português, havia entrada em locales/languages.json mas só servia para instruir o LLM — não tinha arquivo de tradução de UI.

O que foi feito

  1. Criação de locales/pt.json com 646 strings traduzidas a partir de locales/en.json, mantendo todas as chaves originais e placeholders intactos.
  2. Definição de PT-BR como idioma padrão em frontend/src/i18n/index.js:
const savedLocale = localStorage.getItem('locale') || 'pt'   // antes era 'zh'
const i18n = createI18n({
  legacy: false,
  locale: savedLocale,
  fallbackLocale: 'en',                                       // antes era 'zh'
  messages
})
  1. Refinamento da instrução LLM em locales/languages.json:
"pt": {
  "label": "Português (BR)",
  "llmInstruction": "Por favor, responda sempre em português brasileiro
  (pt-BR), usando vocabulário, gramática e expressões naturais do Brasil.
  Não use português europeu."
}
  1. Replicação de regras CSS de html[lang="en"] para html[lang="pt"] usando o seletor moderno :is(), em três arquivos: Home.vue, Step4Report.vue, Step5Interaction.vue. Necessário porque palavras em PT são significativamente mais longas que em chinês.
  2. Edição do frontend/index.html para que o HTML estático já carregue em PT-BR antes do Vue inicializar (lang, title, meta description).
// VALIDAÇÃO ESTRUTURAL Após a tradução, foi rodada comparação automática entre en.json e pt.json: en keys: 646 · pt keys: 646 · missing: 0 · extra: 0 — 100% de paridade.

Como o sistema de i18n funciona internamente

Frontend (Vue + vue-i18n): auto-carrega qualquer arquivo locales/*.json registrado em languages.json. Salva escolha em localStorage sob a chave locale. O componente LanguageSwitcher.vue permite trocar dinamicamente.

Backend (Flask): lê o header HTTP Accept-Language enviado pelo frontend. A função get_language_instruction() retorna o llmInstruction do languages.json para injetar nos prompts do LLM.

O sistema é elegante e plug-and-play: adicionar um novo idioma exige apenas criar o arquivo <codigo>.json e registrar no languages.json. Nenhuma mudança de código backend necessária.

Modus operandi de uma simulação

Os dois eixos independentes

Antes de tudo, é fundamental entender que uma simulação tem dois eixos completamente separados:

EixoO que controlaOnde se ajusta
Quantidade de agentes ("pessoas")Quantas personas o LLM gera (50, 100, 500, 1000+)Implícito: LLM decide pelo prompt + documento. Visualizado no Step 2.
Quantidade de rodadas (ticks)Por quantos ticks o relógio do mundo simulado avançaExplícito: OASIS_DEFAULT_MAX_ROUNDS ou toggle "Personalizado" no Step 2
// ANALOGIAPense em um jogo tipo The Sims. Agentes = quantas pessoas existem na cidade. Rodadas = quantos "dias" o jogo roda antes de você apertar pause.

O que acontece em cada rodada

Cada rodada é um tick de tempo no mundo simulado. Por exemplo, se a simulação representa 24h reais distribuídas em 10 rodadas, cada rodada equivale a ~2,4h do mundo simulado.

  1. Cada agente é "acordado" pelo motor OASIS
  2. O LLM é chamado uma vez por agente para decidir a ação: postar, curtir, comentar, seguir, repostar, ficar quieto
  3. As ações acontecem no Twitter/Reddit simulado (memória no Zep)
  4. Sistema atualiza memórias, conexões sociais, viralidade, tópicos em alta
  5. Avança para a próxima rodada

As 5 etapas (Steps) da UI

STEP 01
GRAFO
Upload + prompt → ontologia → GraphRAG no Zep
STEP 02
SETUP
Personas geradas + config de mundo + ajuste de rodadas
STEP 03
SIMULAR
Loop de M rodadas × N agentes em paralelo (Twitter + Reddit)
STEP 04
REPORT
Report Agent gera relatório multi-seção via ReACT
STEP 05
CHAT
Conversar com agentes / surveys / drill-down

Fórmula de custo e dimensionamento

A fórmula básica

chamadas LLM totais ≈ rodadas × agentes
custo total ≈ chamadas × custo_médio_por_chamada + overhead

O overhead inclui geração de personas, construção de ontologia, configuração de plataforma, planejamento e geração do relatório. Pode somar de 10% a 50% além do cálculo básico.

LLMs locais — viabilidade

A pergunta

"É possível usar LLMs gratuitas que são instaláveis na VPS? Como Llama ou similar?"

A resposta honesta

// VEREDITOTecnicamente sim. Na prática, na sua VPS atual, NÃO.

Por quê?

Sua VPS tem 2 vCPUs, 7,8 GB RAM, sem GPU. LLMs locais minimamente úteis exigem GPU com VRAM ou CPU + muita RAM (16+ GB), e mesmo assim ficam extremamente lentos.

Tabela de tempo por chamada LLM

ModeloHardwareTempo por resposta
Llama 3.2 1BCPU 2-core~10-30s
Llama 3 8B Q4CPU 2-core~60-180s
Qwen-Plus (cloud)API~1-2s
Claude Haiku (cloud)API~1-3s

O efeito disso no MiroFish

2.000 chamadas × 60s = 120.000 segundos = 33 HORAS por simulação

E ainda com qualidade muito pior. Inviável.

Alternativa pragmática: Groq

Se a motivação é "Llama de graça", crie conta no Groq. Free tier generoso, modelos Llama/Mixtral/Gemma rodando em hardware proprietário (~500 tokens/seg).

LLM_BASE_URL=https://api.groq.com/openai/v1
LLM_MODEL_NAME=llama-3.3-70b-versatile
LLM_API_KEY=gsk_...

  • Endpoint base
  • LLM acelerador

  • Default de rodadas (sobrescritível na UI)
  • Modo debug do Flask

Como construir um prompt eficaz

A qualidade da simulação depende 80% da qualidade do prompt + do material-semente. Um prompt vago gera personas genéricas. Um prompt cirúrgico produz insights acionáveis.

Anatomia de um bom prompt MiroFish

01
CONTEXTO
Quem é o público simulado: perfil sócio-demográfico, características, dores
02
ESTÍMULO
O evento, oferta ou material que dispara a simulação
03
VARIÁVEIS
O que pode mudar entre os agentes (heterogeneidade)
04
OBJETIVO
O que você quer prever ou medir
05
RESTRIÇÕES
Limites de tamanho, escopo, tempo simulado

Material-semente: o segundo pilar

O LLM precisa de contexto rico para construir personas realistas. Tipos de material úteis:

  • Roteiros de pitch / scripts comerciais
  • Propostas comerciais completas
  • Pesquisas de mercado / relatórios setoriais
  • Personas de clientes anteriores (mesmo fictícias)
  • Listas de objeções comuns já ouvidas
  • Tabelas de preços + comparativos
  • Briefings de campanha · releases · capítulos
// MÍNIMO VIÁVELSe for subir só 1 documento, consolide os mais importantes em um único PDF de até 30 páginas. Sem material rico, a simulação inventa coisas.

Exemplo: proposta para 200 ISPs

Caso de uso: simular a recepção de uma proposta comercial de serviços de marketing por aproximadamente 200 donos de provedores regionais de internet brasileiros, para prever taxa de fechamento e capturar objeções qualitativas.

O prompt completo

Simule a recepção de uma proposta comercial de serviços de marketing
por aproximadamente 200 donos de provedores regionais de internet (ISPs)
brasileiros, considerando o perfil descrito a seguir.

PERFIL DOS AGENTES (~200 personas):
Donos ou sócios-administradores de provedores de internet de pequeno e
médio porte no Brasil, com 500 a 15.000 assinantes ativos. Distribuídos
por diferentes regiões. Maioria com formação técnica, entre 30 e 55 anos,
autodidatas em gestão, fundadores que cresceram organicamente.
Compartilham as seguintes dores:

- Marketing rudimentar (boca a boca, panfletos, Facebook ads sem estratégia)
- Concorrência crescente de grandes operadoras e ISPs maiores
- Churn alto sem mensuração precisa
- Equipe interna sem skill de marketing digital
- Desconfiança histórica de agências (já foram queimados antes)
- Foco operacional em detrimento do crescimento de receita
- Decisão de compra rápida quando enxergam ROI claro, mas céticos por padrão

Distribua entre eles personas heterogêneas: alguns mais analíticos, outros
mais relacionais, alguns early adopters, outros conservadores. Inclua
variações de humor inicial (cético, neutro, entusiasmado).

ESTÍMULO (evento da simulação):
Cada agente recebeu o pitch comercial em anexo (PDF) e participou de uma
call remota one-to-one de 45 minutos com o vendedor da [SEU NOME / EMPRESA].
Após a call, têm acesso a um "feed simulado" estilo grupo de WhatsApp de
donos de provedores, onde podem comentar a proposta entre si, tirar dúvidas,
compartilhar receios, e influenciar uns aos outros ao longo de ~7 dias
simulados antes de decidir fechar ou não.

VARIÁVEIS QUE QUERO OBSERVAR:
- Quem fecha imediatamente após a call (24h)
- Quem fecha depois de discutir com pares (72h+)
- Quem rejeita imediatamente e por quê
- Quem fica em "vou pensar" indefinido
- Quais objeções aparecem com mais frequência
- Como o sentimento coletivo evolui ao longo dos dias
- Se algum "líder de opinião" emerge no grupo

OBJETIVOS DE PREVISÃO:
1. Estimativa quantitativa de fechamento: % em 7 dias, segmentado
   por porte (pequeno vs médio) e por região
2. Top 5 objeções qualitativas: argumentos contrários e como reagi-los
3. Top 5 ganchos positivos: o que mais ressoou e deve ser amplificado
4. Perfil do comprador ideal (ICP refinado)
5. Sugestões de ajuste no pitch
6. Risco reputacional

HORIZONTE TEMPORAL:
Simule 7 dias de tempo decorrido após a call.

Material-semente recomendado

  1. Sua proposta comercial completa (5-15 páginas)
  2. Roteiro/script do pitch de vendas (transcrição da call de 45 min) — fundamental
  3. Pesquisa de mercado de ISPs brasileiros (ANATEL, Telebrasil, ABRINT)
  4. 3-5 personas de clientes anteriores (mesmo fictícias)
  5. Lista de objeções comuns já ouvidas — ouro puro
  6. Tabela de preços + comparativo com concorrentes

Configuração técnica recomendada

ParâmetroValor sugeridoPor quê
OASIS_DEFAULT_MAX_ROUNDS30 a 507 dias / N rodadas → cada uma representa algumas horas
LLM_MODEL_NAMEqwen-plus ou claude-haiku-4-5Personas críticas precisam de qualidade
LLM_BOOST_MODEL_NAMEqwen-turboModelo barato para os ticks de agente
REPORT_AGENT_TEMPERATURE0.3Relatório analítico, não criativo
REPORT_AGENT_MAX_TOOL_CALLS8Mais investigação por seção

Custo estimado e estratégia

  • Com Qwen-Plus + Qwen-Turbo: US$ 2 a US$ 5
  • Com Claude Haiku via OpenRouter: US$ 8 a US$ 15
// ESTRATÉGIA EM DUAS FASES Fase 1 (validação): rode primeiro com 20 agentes (~US$ 0,20) só para validar que sai sem erro e o relatório serve.
Fase 2 (produção): só depois rode com 200 agentes para valer.

Observações Avançadas

Modo dev vs produção

A imagem custom mirofish-pt:local roda Vite + Flask em modo dev (debug, hot reload, SSE). Para o caso atual está adequado. Para produção pesada, o ideal seria: build estático do Vue, servir os estáticos pelo Nginx, backend atrás de Gunicorn/Uvicorn com workers, FLASK_DEBUG=False. Exigiria customizar o Dockerfile.

Licença AGPL-3.0

O MiroFish é licenciado sob AGPL-3.0. Você pode usar comercialmente e modificar o código. Mas: se expor o serviço publicamente para terceiros, é tecnicamente obrigado a disponibilizar o código modificado — basta manter o link do fork público no GitHub. Como nosso fork douglasemid/MiroFish já está público, essa obrigação está cumprida.

WebSocket / SSE

O frontend Vue se conecta ao backend Flask via:

  • HTTP REST (axios) para criar/listar simulações, upload, comandos
  • Server-Sent Events (SSE) para receber updates em tempo real

Não há Socket.IO ou WebSocket bidirecional — só o canal SSE unidirecional servidor → cliente.

Sistema de filas e tarefas assíncronas

O backend Flask não usa Celery, RQ ou Redis Queue. Tarefas longas rodam em threads internas do Flask com threaded=True. O frontend faz polling em endpoints /api/.../status.

Como o Nginx encaminha o tráfego

Browser → https://miro.emidgroup.com.br
         ↓ (TLS handshake com cert Let's Encrypt)
Nginx (rastelli_nginx, container) — porta 443
         ↓ (server block para miro.emidgroup.com.br)
proxy_pass http://mirofish:3000/        ← frontend Vite
proxy_pass http://mirofish:5001/api/    ← backend Flask
         ↓
Container `mirofish` na rede `app_rastelli-net`

O Nginx usa resolver Docker interno (127.0.0.11) para resolver lazy DNS dos containers. O Host header é enviado como localhost para o frontend (para o Vite aceitar via allowedHosts), e o host real chega no backend via X-Forwarded-Host.

Tradução incompleta — pontos não traduzidos

  • Logs internos do backend Python (ex: MiroFish Backend 启动中...) ainda aparecem em chinês porque são print() hardcoded. Não afetam a UI.
  • Conteúdo dos PDFs/MDs que você sobe é processado no idioma original. Se for inglês, as personas pensam em inglês mas respondem em PT-BR por causa do llmInstruction.
  • Comentários de código dentro dos arquivos JS/Vue/Python permanecem em chinês/inglês — irrelevante para o usuário final.

Troubleshooting comum

Container não inicia / crashloop

Causa mais comum: chave do LLM ou Zep inválida/ausente.

docker logs mirofish 2>&1 | head -30
# Procure por: "LLM_API_KEY 未配置" ou "ZEP_API_KEY 未配置"

502 Bad Gateway no navegador

O container mirofish parou. Verifique:

docker ps --filter name=mirofish
docker logs mirofish 2>&1 | tail -50
cd /home/deploy/mirofish && docker compose up -d

"Vite Blocked request: This host is not allowed"

O Host header não está sendo enviado como localhost. Verifique no nginx.conf se o location / tem:

proxy_set_header Host "localhost";
proxy_set_header X-Forwarded-Host $host;

Disco cheio (100%)

df -h /
docker images
docker image prune -a
